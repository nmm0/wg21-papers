# Motivation and Background

The introduction of `mdspan` in [@P0009R18] raises some issues on how to perform some operation over each element of an `mdspan`.
[@D4152R0] raises the possibility of performing such an algorithm on an `mdspan`.
However, both the existing C++ standard algorithm `for_each` and the proposed `mdspan`-based algorithm have some limitations in usability.

Many algorithms that are useful on `mdspans` look at other elements of the `mdspan`, including elements relative to the current one such as neighbors.
For example, stencil operations in scientific code compute a derivative using neighboring cell elements.
In image processing, convolution operations look at a window surrounding an individual pixel.

Furthermore it is often useful to perform operations on identically-sized containers or `mdspan`s.
For example, in highly-parallel code it is often more efficient to structure code in a structure-of-array (SoA) format, particularly when performing SIMD operations.

For example:

```cpp
auto x = std::vector<double>(vec_size);
auto y = std::vector<double>(vec_size);
auto z = std::vector<double>(vec_size);

for (size_t i = 0; i < vec_size; ++i)
{
  auto foo = compute_something();
  x[i] = compute_x(foo);
  y[i] = compute_y(foo);
  z[i] = compute_z(foo);
}
```

This kind of loop is difficult to do via the existing `for_each` facility, but can be done with traditional indexing-based for loops.
However care must be taken when using a traditional index-based for loop to ensure that the loop nesting follows optimal rank order in order to preserve caching locality; otherwise one may run into significant performance issues.

## Existing practice

### Enumerate

[@P2164R9] introduced `views::enumerate` to C++23. One could use this to write the loop as follows:

```cpp
std::ranges::for_each(x | std::views::enumerate, [&y, &z](auto &&tup) {
  auto &[i, val] = tup;
  auto foo = compute_something();
  val = compute_x(foo);
  y[i] = compute_y(foo);
  z[i] = compute_z(foo);
} );
```

## Iota and Indices

Using `views::enumerate` isn't really ergonomic, but it is an option. Probably more useful is:

```cpp
std::ranges::for_each(std::views::iota(0, vec_size), [&x, &y, &z](auto i) {
  auto foo = compute_something();
  x[i] = compute_x(foo);
  y[i] = compute_y(foo);
  z[i] = compute_z(foo);
} );
```

However, things get more complicated if we want to do multidimensional indexing.
I borrowed the following example from [@P3179R9], which introduced range-based parallel algorithms:

```cpp
std::mdspan A{input,  N, M};
std::mdspan B{output, M, N};

std::ranges::for_each(std::execution::par_unseq,
  std::views::cartesian_product(
    std::views::iota(0, A.extent(0)),
    std::views::iota(0, A.extent(1))),
  [=] (auto idx) {
    auto [i, j] = idx;
    B[j, i] = A[i, j];
  });
```

These two examples actually have a flaw -- if one wants to use `size_t`-based extents (the default) then one must make sure that the first argument to `iota` is a `size_t`, e.g. `views::iota(0uz, A.extent(0))`.

A better alternative is to use the new C++26 feature, `views::indices` introduced in [@P3060R3]:

```cpp
std::ranges::for_each(std::execution::par_unseq,
  std::views::cartesian_product(
    std::views::indices(A.extent(0)),
    std::views::indices(A.extent(1))),
  [=] (auto idx) {
    auto [i, j] = idx;
    B[j, i] = A[i, j];
  });
```

Needless to say, this is still quite verbose.
In particular, the aspects that make this verbose are highlighted as follows:

1. Having to specify `views::cartesian_product` in order to concatenate the dimensions
2. Specifying `views::iota` for each dimension
3. The necessity of decomposing `idx` via structure bindings to obtain the more conventional integer indices -- note that the alternative would be even more verbose (i.e. using `std::get<0>(idx)`)

It's also worth pointing out that if one is accessing an `mdspan` via these multidimensional indices, the layout of the `mdspan` may be inconsistent with the ordering of the cartesian product, leading to inefficient memory caching behavior or other subtle performance issues.
While one could simply reverse the indices in the cartesian product, this can lead to accidental out-of-bounds memory accesses if the user forgets to also reverse the indices inside the loop.

## Traditional index-based for loops

For now, it's likely more simple to just perform a more traditional nested index-based for loop:

```cpp
for (size_t j = 0; j < M; ++j)
  for (size_t i = 0; i < N; ++i)
    B[j, i] = A[i, j];
```

While significantly easier to read, it is still unergonomic to change iteration order as the loops need to be inverted.
Of course, the more glaring issue is that the above loop is more difficult to parallelize, requiring the user to use compiler extensions such as OpenMP [@OpenMP].

## Non-standard approaches

Applications designed for high-performance data processing and scientific computing tend to use index-based looping extensively.
Therefore, many libraries designed for these applications provide facilities for index-based and multidimensional looping.

The Kokkos performance-portability framework provides an `MDRangePolicy` for their `parallel_for` construct (and indeed provides one-dimensional looping with `RangePolicy`) [@KokkosMDRange].

CUB, or the "Cuda Unbounded" library by NVIDIA, provides constructs for iterating on the device with the `cub::DeviceFor` set of functions [@CUBDeviceFor], including ways of iterating over `mdspan` extents and layouts via `cub::DeviceFor::ForEachInExtents` and `cub::DeviceFor::ForEachInLayout`.

HPX, which provides a parallel library that closely mirrors the standard C++ library adds extensions for index-based looping algorithms [@HPXForLoop].

# Proposed Interface

In short, we would like to propose `for_each_index`, an algorithm that enables single and multi-dimensional index-based loops.

Taking the above matrix transpose example, the usage would look something like this:

```cpp
std::for_each_index(std::execution::par_unseq,
  std::layout_right(A.extents()),
  [=] (auto i, auto j) {
    B[j, i] = A[i, j];
  });
```

We propose six overloads for `for_each_index`; two of them are used for single-dimensional indexing and four for multidimensional (though of course a rank-1 extent or layout mapping can still be used for single-dimensional iteration).

Interestingly, rank-0 extents and layout mappings are also possible. These would simply require a function taking no arguments to be passed in as the final argument.
While this may be of limited use, we see little use in preventing this.

```cpp
template<class IntegralExtent, class Fun>
void for_each_index(IntegralExtent extent, Fun &fun);

template<class Extents, class Fun>
void for_each_index(const Extents &exts, Fun &fun);

template<class Mapping, class Fun>
void for_each_index(const Mapping &mapping, Fun &fun);

template<class ExecutionPolicy, class IntegralExtent, class Fun>
void for_each_index(ExecutionPolicy &&policy, IntegralExtent extent, Fun &fun);

template<class ExecutionPolicy, class Extents, class Fun>
void for_each_index(ExecutionPolicy &&policy, const Extents &exts, Fun &fun);

template<class ExecutionPolicy, class Mapping, class Fun>
void for_each_index(ExecutionPolicy &&policy, const Mapping &mapping, Fun &fun);
```

## Iteration order

The central design here beyond single and multi-dimensional indexing is the possibility of providing a layout mapping to `for_each_index`.
This provides a hint to the implementation of the best order to traverse the index space in order to optimize loop performance.
Typically, if an `mdspan` is accessed inside the loop, the intent is for the user to provide the layout mapping of the mdspan in order to get the best performance out of the loop.

The strategy that the implementation actually uses should be implementation defined.
For example, one strategy may be to iterate over each rank in the order of the smallest to largest stride, in order to maximize cache reuse.
However, we would like to avoid over-constraining an implementation and our hope is that a high quality implementation will actually provide good performance.

While the NVIDIA CUB library [@CUBDeviceFor] only permits `layout_left` and `layout_right`, we plan on allowing any arbitrary layouts including custom layouts.
Likely an implementation can use information about the layout mapping's strides or other properties (such as `is_exhaustive` and `is_unique`) in order to plan an optimal scheduling or iteration strategy.
It's worth noting that because of the unlimited customization possibilities of layout mappings, implementations may not always be able to pick the most optimal strategy.


# Design Considerations

## Rank-1 integer overload

This paper proposes a rank-1 overload that takes a nonnegative integer type as its first non-execution policy parameter.
The authors disagree on whether this should be included as the alternative is only slightly more verbose, e.g.:

::: cmptable

### Range-based

```cpp
ranges::for_each(views::indices(N), fun);
```

### Index-based

```cpp
for_each_index(N, fun);
```

:::

Nevertheless, the authors are in agreement that if this overload is provided it should be in the `<algorithm>` header.

## Header

We chose `<mdalgorithm>` as a header fot the multidimensional `for_each_index` overloads to keeping the naming convention from `<mdspan>`.
There was some discussion on whether to include these in either `<mdspan>` or `<algorithm>`.
We were against this because in particular the multidimensional `for_each_index` algorithm can be quite heavy depending on the implementation strategy and thus we would not want to include this in `<mdspan>`.
This argument also applies to `<algorithm>`, but moreover we would also need to pull in the `<mdspan>` header.
Finally, as we plan on proposing more multidimensional algorithms we wanted to create a header for this.

## Alternative Names

* `for_each_in_extents`/`for_each_in_layout_mapping` -- We don't see any need to differentiate the overloads
* `for_each` -- These do slightly different things, this may be confusing as the usage pattern is different (as the non-range `for_each` takes an iterator pair)

## Should we make layout mappings and extents ranges and just use `ranges::for_each`?

No.

But more seriously it's really stretching the definition of a range here, and moreover is still not as ergonomic as you need to still would get a tuple of indices.

## Does this supersede the need for a `for_each` algorithm for `mdspan` proposed in [@D4152R0]?

No, if operating only on one `mdspan`, an implementation can perform additional optimizations by only using the linear indices of an `mdspan`.
Calculating multidimensional indices carry some overhead so if the intended usage allows one to avoid the additional cost then that algorithm may be preferable.

## Do we want to return anything?

We are not entirely sure how useful this would be, but don't have a particular position on it.
