# Motivation and Background

[@P0009R18] introduced to C++ the `mdspan` type, a non-owning multidimensional view.
A distinct property of `mdspan` is that it has a customizable layout that determines how multidimensional indices are mapped to offsets off the `mdspan`'s internal data handle.
Furthormore, `mdspan`s may have customized accessors; these determine how `mdspan` obtains references to data from offsets.
The standard `std::default_accessor` simply dereferences the offset data handle, but custom accessors could generate proxy references or even perform accesses over a network.

A user of `mdspan` may find themselves wanting to apply some of C++'s standard algorithms to an `mdspan`.
However, they would find some difficulty; `mdspan` does not provide `begin()` or `end()` iterators (nor range access).
Furthermore, it is not clear what exactly these iterators would be!

Of course you could imagine what a trivial `mdspan`'s begin and end iterators would be; they could be simply `mds.data_handle()` and `mds.data_handle() + mds.size()`.
For anything more complex than a contiguous `mdspan` with pointer-based `data_handle`s however, we quickly run into problems.
One might need to wrap `data_handle` in some type of iterator; the iterator dereference would just call `access()` on the `mdspan` accessor.
However, iterating is less straightforward.

This is because the index provided to an `mdspan`'s accessor is **not** constrained in any way to be between `0` and `mds.size()`.
So for example, take this (admittedly contrived) mapping and accessor:

```cpp
class weird_accessor {
  // ...
  // Precondition: i >= 1000
  constexpr reference access(data_handle_type p, size_t i) const noexcept {
    return p[i - 1000];
  }
  // ...
};

// mapping
class weird_layout::mapping : public layout_left::mapping {
  // ...
  template<class... Indices>
  constexpr index_type operator()(Indices... i) const noexcept {
    return layout_left::mapping::operator()(i...) + 1000;
  }
  // ...
};
```

This would rule out trying to iterate in the 1-dimensional space in $\left[0, size\right)$.

The only correct way to iterate over every element in an `mdspan` then, is to traverse every element in every extent of every rank.
This now leads to some ambiguity.
Which rank shall be the fastest varying?
What direction is the most useful to step through that rank?
The wrong choice here could lead to poor performance and caching behavior.

Therefore, this paper proposes that standard algorithms (and their associated parallel versions) should be adapted for `mdspan`s, and that operate on whole `mdspan`s rather than iterators or ranges.

# General Design Considerations

There is some consideration about which standard algorithms can even be adapted for `mdspan`s.
For example -- the algorithms that involve removing elements do not seem conceptually appropriate for an `mdspan`.

On the other hand there remains the question of whether we should adapt algorithms that imply some sort of ordering?
For example, what does it mean to apply `reverse` to an `mdspan`?

Interestingly, if we examine the case where the `mdspan` has a rank of 1, then algorithms that operate on sequences now make sense.
This is also implies that `std::begin` and `std::end` could be applied on an `mdspan` of rank 1.
Supposing we allow these sorts of algorithms on rank-1 `mdspan`s, these could also be useful for `mdspan`s with rank $> 1$.
One can use `submdspan` to obtain e.g. a row or column from a higher rank `mdspan`.
Essentially, this means that if these sorts of algorithms are allowed on rank-1 `mdspan`s, one could perform them on a row or column.

For example, lets say that we have an mdspan representing a matrix (and therefore is rank 2).
We know our matrix is sparse (i.e. many of the entries are $0$), and we would like to compress them using the CRS representation.
So assuming we could allow `transform_exclusive_scan` on a rank-1 `mdspan` we could create our CRS representation elegantly:

```cpp
mdspan<double, dims<2>> matrix = /* ... */;

auto index_if_nonzero = [](double v) { return static_cast<size_t>(v != 0.0); };

const size_t count = count_if(matrix, index_if_nonzero);
vector<double> values(count);
vector<size_t> col_indices(count);
vector<size_t> row_indices(matrix.extents().extent(0) + 1)
vector<size_t> indices(matrix.extents().extent(1));
size_t offset = 0;
for (size_t r = 0; r < matrix.extents().extent(0); ++r)
{
  std::transform_exclusive_scan(submdspan(matrix, r, full_extent), indices.begin(), 0uz, std::plus<size_t>{}, index_if_nonzero);

  const auto count_in_row = indices[indices.size() - 1];
  
  for (size_t c = 0; c < matrix.extents().extent(1); ++c)
  {
    auto v = matrix(r, c);
    if (index_if_nonzero(v))
    {
      values[offset + indices[c]] = v;
      col_indices[offset + indices[c]] = c;
    }
  }

  row_indices[r] = offset;
  offset += count_in_row;
}

row_indices[row_indices.size() - 1] = offset;
```

There are two main design questions then:

1. Should we permit sequence algorithms to apply to rank-1 `mdspan`s?
2. If so, should we add support for `begin` and `end` to rank-1 `mdspan`s?

It's worth mentioning that if we allow `begin` and `end` to work with rank-1 `mdspan`s, we could also choose to just let users use the existing range-based or iterator-based algorithms rather than adding rank-1 mdspan overloads.

## Why not allow only whole mdspans and not subsets of elements in mdspans?

This is actually a trick queestion.
This paper actually does allow partial mdspans to be used in an algorithm.
`submdspan` after all, yields an `mdspan` (usually with a different layout) and thus can be used to create `mdspan`s for use in these algorithms.

One interesting benefit of this is safety; `submdspan` already has preconditions; if we assume any `mdspan` we pass is a valid one, we cannot get out of bounds access during the execution of one of these algorithms.
